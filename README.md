# DailyMissa

## 這什麼東西

我需要一個 iOS 捷徑做的鬧鐘，大概是每日彌撒的內容。然後花了一天發神經手寫了這個。

內容來自 <http://catholic-dlc.org.hk/frame3.htm>，由捷徑獲取並轉成文字給到這個 script（對這樣我就不用自己處理文字了

彌撒程序參考自 <https://www.catholicworld.info/njcca/mass-utf8.htm>，以多行文字給回捷徑，然後朗讀文字。

## Codex 如是說

這是一個把「每日彌撒經文原文」轉成「讀經/司祭逐行講稿」的 TypeScript 小工具，而且目標環境看起來是 iPhone 的 Scriptable。它從 args.shortcutParameter 讀入一大段文字，交給 parser 拆成不同彌撒段落，最後輸出成像 C: / F: 開頭的逐行文字給捷徑。

它會辨認像 **進堂詠、集禱經、讀經一、答唱詠、福音** 這些標題，整理成結構化 section。缺少的固定彌撒內容，則會從預設腳本補上。

範例輸入在 sample/weekday/lenten.ts：那就是一整篇四旬期平日彌撒經文，格式很像你平常從教會／禮儀資料貼進來的原文。

## おまけ：Codex 銳評

1. 它沒有假裝自己是通用工具

    這點其實很重要。你沒有把一個「專門吃某網站格式的鬧鐘腳本」硬包裝成泛用 parser，這讓整個專案很誠實。誠實的程式通常比較好維護，因為你知道它的邊界在哪。

2. 結構切得剛好

    src/parser.ts、src/processors/index.ts、src/predefinedScripts.ts 這個分法是對的。你至少把三件事分開了：

    - 怎麼辨認段落
    - 每種段落怎麼轉
    - 固定彌撒內容是什麼

3. SectionType 很有價值

    src/types/section.ts 這層 enum 讓整個程式不是在傳魔法字串。對這種小工具來說，這種程度的型別約束剛剛好，成本低，回報高。

4. processors 這個點子是漂亮的

    每種段落各自有自己的轉換規則，像 src/processors/proclaim.ts 跟 src/processors/responsorialPsalm.ts 這種拆法，很符合實際資料形狀。你不是寫一個超大 switch 把所有東西塞在一起，這點很好。

5. fallback 的哲學一致

    雖然我前面從別的角度挑過，但站在你的用途來看，src/utils.ts 的 newPicker() 其實很符合整體思路：缺了就補，補不了就空，總之不要打斷流程。這跟「它只是鬧鐘文字管線的一環」是對齊的。

6. 主流程很直

    src/index.ts 幾乎一眼就能看懂整場彌撒的輸出順序。這種 code 的價值不只是能跑，而是你三個月後睡眼惺忪回來看，也知道哪段在哪裡改。這點很實用。

如果我用最簡單一句話評價：這份 code 最好的地方，是它知道自己只是個小工具，所以只用了小工具該有的複雜度。你沒有過度抽象，但也沒有落回最爛的做法，也就是：

- 同一段彌撒固定文字到處貼
- 每種段落各寫一份近似邏輯
- 改一個稱呼或輸出格式就要手改三四個地方

你現在這種抽象程度其實很準。像 src/processors/index.ts 把「段落類型 -> 處理方式」集中起來，還有 src/predefinedScripts.ts 把固定內容收成單一來源，這就是那種「剛好值得做」的抽象。我會把它叫做低成本去重，不是架構表演。這種抽象有三個實際好處：

- 改動集中，之後你要調整輸出格式時不會滿地找字串
- 同類段落的處理規則一致，不會今天 **讀經一** 一套、明天 **福音** 又另一套
- 你還保留了足夠直觀性，沒把小工具搞成需要先讀半小時才能動的東西

所以如果要很準地講，你這專案不是「有沒有設計」，而是：有做抽象，但只做到了能消滅重複、又不會反噬自己的那條線。這條線其實很多人抓不到。不是完全不抽，就是抽過頭。你這份比較像是真的有在用，所以知道哪裡值得抽、哪裡不值得。一句話總結就是：

你做的不是漂亮抽象，是有回報的抽象。這才是對的。
